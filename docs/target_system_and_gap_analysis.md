# 目标系统与差距分析

## 1. 目标系统定位

AgentGuard 的目标是实现一个面向大模型 Agent 的工具调用安全评测与防护平台。系统应覆盖 Agent 调用文件、数据库、代码执行、API、知识库等外部工具时可能出现的提示注入、越权访问、敏感信息泄露和高危操作失控等问题。

项目定位为研究型原型，不追求工业级复杂度，但必须做到：

- 可运行：Agent 能完成基本工具调用任务。
- 可复现：评测集、实验脚本和指标输出稳定。
- 可展示：能通过页面或命令行展示调用链、风险判断和拦截原因。
- 可量化：能对比不同防护方案的任务完成率和安全指标。

## 2. 目标系统组成

目标系统应包含六个核心模块：

| 模块 | 目标功能 |
|---|---|
| 基础 Agent 系统 | 接收自然语言任务，完成文件读取、数据库查询、知识库检索、报告生成等典型流程 |
| 工具注册中心 | 为每类工具配置操作类型、风险等级、权限范围、参数约束和人工确认策略 |
| 安全策略引擎 | 在工具执行前完成权限校验、参数检查、风险分级、敏感信息检测和高危操作判断 |
| 安全网关 | 根据策略结果决定允许、阻断、脱敏或要求人工确认 |
| 评测与实验模块 | 构建安全评测集，对比无防护、提示词约束、规则防护和安全网关方案 |
| 审计与可视化 | 展示每一步工具调用、风险判断、拦截原因和实验结果 |

目标调用链如下：

```text
用户任务
  -> Agent 规划并生成工具调用
  -> 工具注册中心提供工具策略
  -> 安全策略引擎检查风险
  -> 安全网关决策
  -> 工具执行或阻断
  -> 审计日志记录
  -> 可视化页面展示
```

## 3. 当前已完成内容

当前原型已经完成了安全评测与防护平台的主要后端能力：

- 工具注册中心：已通过 `data/tools.json` 配置文件、数据库、代码执行、API、搜索等工具的风险等级、权限和参数约束。
- 安全网关：已支持工具注册检查、角色权限校验、scope 校验、路径约束、SQL 只读检查、URL 白名单、敏感信息检测、提示注入检测、高危工具确认和输出脱敏。
- 评测集：已覆盖正常任务、直接提示注入、间接提示注入、越权访问、敏感泄露、高危工具调用和参数篡改。
- 自动化评测：已支持 `none`、`prompt_only`、`rule_guard`、`gateway` 四种方案对比。
- 审计日志：已能记录工具调用、上下文、风险信号、策略决策和执行结果。
- 实验结果：已能输出 `metrics.json`、`report.md` 和各模式审计日志。
- 项目文档：已包含风险分类、系统设计、实验报告和演示指南。

## 4. 当前主要缺口

### 4.1 缺少基础 Agent 执行循环

当前评测主要基于预设工具调用轨迹，还没有一个从自然语言任务出发、自动选择工具并执行多步任务的 Agent。

需要补充：

- `DemoAgent` 或 `RuleBasedAgent`。
- 自然语言任务入口。
- 简单任务规划与工具选择。
- “文件读取 -> 数据查询 -> 知识库检索 -> 报告生成”的完整演示链路。

优先级：高。

### 4.2 缺少知识库检索工具

当前已有 mock 搜索工具，但没有明确的知识库工具。

需要补充：

- `kb.search` 工具。
- 本地知识库目录，例如 `data/demo_workspace/kb/`。
- 知识库正常检索样本。
- 知识库间接提示注入样本。

优先级：高。

### 4.3 报告生成链路不完整

当前有 `file.write`，但报告生成仍是单步工具调用，不是 Agent 汇总多工具结果后的完整任务。

需要补充一个典型演示流程：

```text
读取公开项目说明
  -> 查询开放风险项
  -> 检索知识库建议
  -> 生成安全评估简报
  -> 写入 scratch/report.md
```

优先级：高。

### 4.4 缺少简单可视化页面

当前只有 CLI、Markdown 报告和 JSONL 审计日志，还没有页面化展示。

需要补充：

- 实验指标面板。
- 审计日志列表。
- 单任务调用链回放。
- 风险等级、决策结果和拦截原因展示。
- 可选的高危操作确认按钮。

优先级：高。

### 4.5 高危操作确认仍停留在决策状态

当前网关能返回 `require_confirmation`，但缺少完整确认流程。

需要补充：

- 确认请求展示。
- 确认或拒绝操作。
- 确认后重新执行工具调用。
- 审计日志记录确认动作。

优先级：中。

### 4.6 策略引擎模块边界还不够清晰

当前策略判断主要集中在安全网关内部，功能已经具备，但汇报时“策略引擎”这个模块不够显式。

需要补充：

- 独立的 `policy_engine.py` 或策略组件。
- 将权限策略、参数策略、敏感信息策略、提示注入策略和高危策略拆分为更清晰的检查单元。

优先级：中。

## 5. 建议补齐顺序

建议按以下顺序继续开发：

1. 实现基础 Agent 执行循环。
2. 增加知识库检索工具 `kb.search`。
3. 增加完整报告生成任务链路。
4. 实现简单可视化页面。
5. 增加高危操作确认交互。
6. 将策略引擎模块显式化。
7. 优化中文实验报告和一键演示脚本。

## 6. 六个月项目边界

六个月内建议聚焦研究原型，不做工业级平台。

应重点完成：

- Agent 基本任务执行。
- 工具注册与权限策略。
- 安全网关真实拦截风险调用。
- 主要风险场景评测集。
- 自动化实验和可量化指标。
- 审计日志和简单可视化展示。

可以简化：

- Agent 可使用规则或 Mock LLM。
- 工具可使用本地模拟实现。
- 策略引擎可使用轻量规则，不必引入复杂策略语言。
- 审计日志可使用 JSONL，不必做不可篡改日志链。

作为未来工作：

- 真实大模型 Agent 接入。
- 大规模评测集。
- 多模型对比。
- 容器沙箱。
- 企业级策略语言。
- 生产部署和完整权限后台。

## 7. 总结

当前 AgentGuard 已经完成安全网关、工具注册、评测集、自动化实验、审计日志和主要文档，能够证明运行时安全网关对不安全工具调用有明显防护效果。

按照目标要求，当前最需要补的是两件事：

- 基础 Agent 系统：让项目从“评测轨迹”升级为“Agent 可执行任务”。
- 简单可视化页面：让调用链、风险判断、拦截原因和实验结果更容易展示。

补齐这两部分后，项目会更符合“可运行、可复现、可展示、可量化”的完整要求。

## 8. 本次补齐状态

已按上述差距补齐研究型原型所需的关键模块：

- 基础 Agent 系统：新增 `agentguard/agents/DemoAgent`，支持自然语言任务入口和“文件读取 -> 数据查询 -> 知识库检索 -> 报告生成”的多步链路。

- 审计与可视化：新增 `agentguard/ui` 静态仪表盘生成器，可展示指标、审计日志、调用链、风险等级、决策原因和确认按钮。
- 高危确认流程：新增 `SecurityGateway.resolve_confirmation` 和 `confirm-demo` CLI，支持确认请求、拒绝或确认后重放执行并写入审计日志。

## 9. 进一步 Agent 场景化补强

为了减少“demo”感，系统新增了更具体的受测 Agent：

- `SecurityOperationsAgent`：面向 SOC 告警研判场景，采用 planner-executor 工作流，能够解析告警编号、查询告警和资产上下文、调用威胁情报、检索处置知识库、隔离投毒内容并生成研判报告。
- `data/security_ops_workspace/`：新增 SOC 章程、告警 intake、处置 playbook、投毒 vendor advisory、私有 token 文件和 reports 目录。
- `threat.lookup`：新增本地威胁情报工具，用于模拟真实安全运营 Agent 的外部情报调用。
- SOC 攻击样本：新增 vendor advisory 间接提示注入和 threat-intel secret leakage 场景。
- CLI：新增 `security-agent` 命令，作为当前主要受测 Agent 入口；旧 `agent` 命令保留为轻量兼容演示。
